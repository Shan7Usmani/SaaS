## Key Structural Decisions

| Decision | Rationale |
|----------|-----------|
| **Route groups** `(auth)`, `(dashboard)`, `(marketing)` | Separate layouts per auth state without URL prefix pollution |
| **`/api` colocated in app dir** | Next.js Route Handlers are colocated with the app вҖ” no separate API server needed for MVP |
| **Components split by feature** | Each module gets its own folder. Prevents cross-contamination. Easy to tree-shake when deferred. |
| **`lib/ai/prompts/`** | Prompts versioned like code. Each AI feature has its own prompt file. |
| **`lib/validators/`** | Zod schemas separated from route handlers for testability |
| **`lib/supabase/` 3 clients** | Browser client (anon key), server client (user JWT), admin client (service role) |
| **`types/` flat** | Types are thin вҖ” no deep nesting. Supabase-generated types live here. |
| **`supabase/` at root** | Keeps DB schema out of app code. Database and app evolve independently. |
| **`dsa/`, `code-mentor/` nested** | Post-MVP components pre-emptively structured but not imported in MVP routes |
